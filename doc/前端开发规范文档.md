# AI短视频自动发布智能体前端开发规范文档

**项目名称：** EasyShorts AI / AI短视频自动发布智能体  
**文档版本：** V1.0  
**适用范围：** Web管理后台前端开发  
**编写日期：** 2026-04-14

---

## 一、文档目的

本文档基于项目现有《需求文档》与《技术架构文档》整理，目标是统一本项目的前端研发方式、页面设计边界、代码组织、接口协作、质量要求与交付标准，确保前端实现具备以下特征：

1. 贴合项目业务目标，服务于“新闻采集、内容生成、视频审核、自动发布、数据分析、系统配置”全流程运营。
2. 与既定技术架构保持一致，遵循 `Vue 3 + Vite + TypeScript + Element Plus` 技术路线。
3. 支持后台系统常见的高可维护、高一致性、高扩展性开发模式。
4. 为后续新增平台、新增任务类型、新增数据维度提供稳定的扩展基础。

## 二、项目定位与前端职责

### 2.1 前端产品定位

本项目前端为**运营管理后台**，不是面向普通消费者的内容浏览站点。前端核心价值在于帮助运营者、审核者和管理员完成以下工作：

1. 查看与管理AI新闻素材。
2. 配置内容筛选、Prompt模板、数据源、平台账号等系统参数。
3. 发起、跟踪、干预和审核视频生成任务。
4. 审核视频、字幕、标题、标签等发布内容。
5. 管理自动发布流程与查看发布结果。
6. 统计平台数据并进行效果分析。

### 2.2 前端职责边界

前端负责：

1. 后台页面展示与交互。
2. 表单输入、列表查询、状态可视化、任务追踪、权限控制。
3. 与 FastAPI 提供的 RESTful API 进行交互。
4. 文件上传、媒资预览、视频审核与发布确认。
5. 数据图表展示、错误提示、操作反馈与审计信息展示。

前端不负责：

1. Agent编排逻辑本身。
2. 视频生成、语音合成、抓取调度等服务端执行逻辑。
3. 业务安全规则的最终裁定。
4. 数据真实性校验的最终责任。

3. 视频任务列表与视频预览。
4. 抖音发布表单与发布记录查询。
5. 定时任务状态展示。
6. 基础系统配置页面。

### 3.2 二期建议扩展

1. 快手平台独立管理能力。
2. 多维度数据分析看板。
3. 审核工作流与人工干预流转。
4. 多语言内容管理。
5. 多平台统一发布编排中心。

## 四、前端技术栈规范

### 4.1 必选技术栈

本项目必须采用以下技术栈：

1. `Vue 3`：前端核心框架，统一使用 Composition API。
2. `TypeScript`：所有业务代码必须使用 TypeScript 编写。
3. `Vite`：本地开发与构建工具。
4. `Element Plus`：后台管理界面基础组件库。
5. `Vue Router`：路由管理。
6. `Pinia`：全局状态管理。

### 4.2 推荐技术栈

在不破坏现有架构的前提下，推荐使用：

1. `Axios`：HTTP 请求封装。
2. `ECharts`：数据分析图表展示。
3. `Sass/SCSS`：样式组织与主题扩展。
4. `ESLint + Prettier`：代码质量与格式统一。
5. `Husky + lint-staged`：提交前校验。
6. `Vitest`：单元测试。
7. `Playwright`：关键页面和流程的端到端验证。

### 4.3 技术使用原则

1. 优先复用框架与组件库能力，不重复造轮子。
2. 优先遵循项目约定的封装层级，不跨层直接访问底层能力。
3. 非必要不引入新依赖；确需引入时，必须说明解决的问题、替代方案和维护成本。
4. 所有新增技术方案需与当前后台系统场景匹配，避免为了展示而过度设计。

## 五、页面与路由规范

### 5.1 推荐路由结构

```text
/login
/403
/404
/dashboard
/news/list
/news/detail/:id
/content/scripts
/video/tasks
/video/detail/:id
/publish/create
/publish/history
/analytics/overview
/analytics/report
/accounts/platform
/system/prompts
/system/sources
/system/tasks
/system/logs
/system/users
/system/roles
```

### 5.2 路由设计原则

1. 一级路由对应一级业务域。
2. 二级路由对应具体业务页面，不允许随意堆叠超过三级页面深度。
3. 详情页统一采用 `:id` 动态参数。
4. 查询条件优先通过 URL Query 保存，保证列表页可分享、可刷新恢复。
5. 路由 `meta` 必须维护标题、权限标识、缓存策略、面包屑信息。

### 5.3 页面类型规范

后台页面统一分为以下类型：

1. 列表页：查询、筛选、分页、批量操作。
2. 详情页：完整信息展示、时间线、日志、操作记录。
3. 表单页：新建、编辑、参数配置、发布设置。
4. 看板页：图表、概览卡片、趋势信息。
5. 任务页：异步状态、进度追踪、失败重试。

## 六、目录结构规范

推荐目录结构如下：

```text
src/
  api/                  # 接口定义与请求封装
  assets/               # 静态资源
  components/           # 全局通用组件
  constants/            # 常量、枚举、映射关系
  directives/           # 自定义指令
  hooks/                # 通用组合式逻辑
  layout/               # 布局组件
  router/               # 路由定义与守卫
  stores/               # Pinia 状态
  styles/               # 全局样式、变量、主题
  types/                # 全局类型定义
  utils/                # 工具函数
  views/                # 页面级模块
    dashboard/
    news/
    content/
    video/
    publish/
    analytics/
    accounts/
    system/
  App.vue
  main.ts
```

### 6.1 目录使用原则

1. 页面级逻辑放入 `views/`，通用逻辑不得反向依赖页面目录。
2. 可复用业务组件优先放在各业务模块内部的 `components/` 子目录，真正跨模块复用的组件再提升到全局 `components/`。
3. 接口类型与接口方法优先按业务域组织，不允许所有接口堆在单一文件中。
4. 工具函数必须保持纯函数优先，避免掺杂 UI 行为。

## 七、编码规范

### 7.1 通用要求

1. 禁止使用 `any` 作为默认兜底类型，确需使用时必须说明原因。
2. 所有接口响应、表单模型、状态对象都必须定义明确类型。
3. 统一使用 Composition API 与 `script setup`。
4. 单文件组件命名使用 `PascalCase`。
5. 文件命名遵循语义化，不使用 `test.vue`、`index2.ts` 等无意义名称。

### 7.2 组件规范

1. 一个组件只负责一个明确职责。
2. 页面组件负责页面编排，通用业务逻辑应抽离到 `hooks` 或业务组件中。
3. 子组件通过 `props` 与 `emit` 通信，避免隐式依赖父级状态。
4. 大型弹窗、复杂表单、表格工具栏建议拆为独立组件。
5. 组件必须考虑空状态、加载状态、错误状态和禁用状态。

### 7.3 TypeScript规范

1. 类型命名使用 `PascalCase`，接口类型建议使用 `xxxItem`、`xxxDetail`、`xxxParams`、`xxxResponse` 等清晰后缀。
2. 枚举值优先与后端约定保持一致，展示文案通过映射表转换。
3. 后端返回状态值不得直接写死在模板中，应统一经常量或格式化函数处理。
4. 所有日期、状态、平台类型、任务类型必须使用统一枚举定义。

### 7.4 样式规范

1. 优先使用组件库能力与项目全局样式变量。
2. 样式类名建议使用语义化命名，避免纯表现型命名。
3. 不允许在页面中堆叠大量内联样式。
4. 同一业务模块的间距、字号、色值、边框、圆角需统一。
5. 后台系统默认遵循简洁、清晰、稳定的视觉原则，不做与业务无关的装饰性设计。

## 八、UI与交互规范

### 8.1 总体交互原则

1. 优先保证信息清晰、操作稳定、反馈及时。
2. 任务型页面优先突出状态、结果、下一步动作。
3. 用户每次操作后必须获得明确反馈，包括成功、失败、处理中三类结果。
4. 异步长任务必须展示进度、当前状态、开始时间、最近更新时间和失败原因。

### 8.2 列表页规范

列表页建议统一包含以下区域：

1. 页面标题区。
2. 查询筛选区。
3. 操作区。
4. 数据表格区。
5. 分页区。

列表页统一要求：

1. 默认提供分页，不使用无限滚动替代后台列表。
2. 查询条件超过三个时，应支持展开/收起。
3. 批量操作必须在选中项后才可用。
4. 危险操作必须二次确认。
5. 表格列内容过长时使用省略与悬浮提示，不破坏布局。

### 8.3 表单页规范

1. 表单项顺序遵循业务流程与用户心智。
2. 必填项必须明确标记。
3. 提交前必须执行前端校验，提交后需结合服务端结果展示错误。
4. 复杂配置表单按分组或分区块组织。
5. 配置保存成功后应明确提示，并保留可追踪的更新时间。

### 8.4 任务状态规范

统一任务状态建议如下：

| 状态值 | 展示文案 | 说明 |
| --- | --- | --- |
| pending | 待执行 | 已创建，等待调度 |
| running | 执行中 | 正在处理中 |
| success | 成功 | 已完成 |
| failed | 失败 | 执行失败，可查看原因 |
| canceled | 已取消 | 被用户或系统取消 |

统一要求：

1. 状态颜色、图标、标签风格全局统一。
2. 失败状态必须展示摘要原因，并支持查看完整日志。
3. 长任务页面必须支持手动刷新或自动轮询。
4. 有重试语义的任务应提供“重新执行”操作。

### 8.5 视频审核与发布交互规范

1. 视频预览、标题、标签、脚本、字幕、发布时间必须在同一页面内可连续处理。
2. 发布前必须展示平台、账号、发布时间、标题、标签、视频源等关键确认信息。
3. 若平台授权失效，必须阻断发布操作并给出明确修复指引。
4. 自动发布和人工发布要有显式区分。
5. 发布成功后应显示发布平台、发布时间、任务编号和结果状态。

## 九、接口协作规范

### 9.1 请求封装规范

1. 所有 HTTP 请求统一通过 `api/` 层封装，不允许在页面组件中直接书写裸请求。
2. 统一处理 `baseURL`、超时、鉴权头、错误拦截、重试策略。
3. 文件上传、下载、流式响应、轮询请求应单独封装。
4. 请求参数对象与响应对象必须具备明确 TypeScript 类型。

### 9.2 接口命名规范

推荐按业务域划分：

```text
api/news.ts
api/content.ts
api/video.ts
api/publish.ts
api/analytics.ts
api/account.ts
api/system.ts
api/auth.ts
```

### 9.3 响应处理规范

1. 统一判断业务状态码与 HTTP 状态码。
2. 统一错误提示文案，不将后端原始异常直接暴露给终端用户。
3. 对于表单校验类错误，应定位到字段级反馈。
4. 对于系统级错误，应保留错误编号、请求时间和建议动作。

### 9.4 长任务处理规范

由于本项目存在新闻抓取、脚本生成、视频合成、自动发布等异步任务，前端必须支持：

1. 任务创建后立即返回任务ID。
2. 基于任务ID轮询任务状态。
3. 页面卸载时自动清理轮询。
4. 长时间运行任务的友好提示。
5. 失败任务展示错误信息与重试入口。

## 十、状态管理规范

### 10.1 Pinia使用边界

适合放入全局状态的内容：

1. 用户信息。
2. 登录态与权限信息。
3. 菜单、路由、系统主题。
4. 全局通知数或全局筛选条件。

不应放入全局状态的内容：

1. 某个页面一次性表单草稿。
2. 局部弹窗显隐。
3. 仅单页使用的列表查询条件。

### 10.2 Store设计规范

1. 一个 Store 对应一个明确领域。
2. Store 中避免直接写复杂页面逻辑。
3. 异步行为命名统一为动词形式，如 `fetchList`、`fetchDetail`、`createTask`。
4. 派生数据优先通过 `computed` 或 getter 组织。

## 十一、权限与安全规范

结合技术文档中的 `JWT + RBAC` 设计，前端必须落实以下规范：

### 11.1 登录与鉴权

1. 登录成功后统一保存 Token、用户信息和权限标识。
2. 所有受保护页面必须经过路由守卫校验。
3. Token 失效时应统一处理跳转登录，并保留来源页。
4. 不得将敏感密钥写入前端代码、配置文件或日志。

### 11.2 权限控制

1. 路由级权限：没有权限的页面不可访问。
2. 按钮级权限：没有权限的操作按钮不可见或不可用。
3. 数据级权限若由后端控制，前端需正确处理空结果与提示。
4. 高风险操作如“立即发布”“删除任务”“重试执行”必须具备权限校验和确认弹窗。

### 11.3 安全展示

1. 用户输入内容在展示前必须避免 XSS 风险。
2. 外链必须标明跳转性质。
3. 敏感字段如密钥、Token、账号信息应脱敏展示。
4. 错误日志中不得回显完整凭证信息。

---

## 十二、媒体与文件处理规范

本项目涉及视频、音频、字幕、封面、OSS 文件地址等媒资，前端需遵循以下规范：

1. 上传前校验文件格式、大小和数量限制。
2. 视频预览必须支持加载中、失败占位与地址失效提示。
3. OSS 文件 URL 只作为资源访问地址，不作为业务主键。
4. 字幕、封面、配音结果建议分区展示，避免混在单一表单中降低可读性。
5. 文件上传成功后，应以业务对象ID作为后续关联依据。

## 十三、数据可视化规范

### 13.1 图表使用原则

1. 图表服务于业务判断，不为了展示而堆叠图形。
2. 同一页面图表不宜过多，避免信息噪音。
3. 图表必须配套筛选条件、时间范围和指标说明。
4. 图表颜色、图例、单位、时间轴格式需统一。

### 13.2 推荐图表类型

1. 播放量趋势：折线图。
2. 平台对比：柱状图。
3. 发布状态分布：饼图或环图。
4. 热点来源占比：柱状图或条形图。
5. 任务处理时长：折线图或区间图。

---

## 十四、错误处理与反馈规范

### 14.1 错误分级

1. 输入错误：字段级提示。
2. 业务错误：页面内提示或消息提示。
3. 系统错误：统一异常提示页或错误弹窗。
4. 网络错误：提示用户重试并保留当前上下文。

### 14.2 反馈规范

1. 提交操作使用按钮 loading，防止重复提交。
2. 删除、发布、重试等关键操作必须反馈结果。
3. 批量操作完成后需展示成功数、失败数和失败原因入口。
4. 页面首次加载失败应支持重新加载。

---

## 十五、性能规范

结合项目指标与后台系统特征，前端需满足以下要求：

1. 首屏核心框架加载稳定，避免因大体积依赖导致后台首开过慢。
2. 路由级页面组件按需加载。
3. 大表格页面需控制一次性渲染量，必要时引入虚拟滚动方案。
4. 图表组件按需加载，避免非图表页面引入大包体。
5. 高频轮询接口应可配置轮询间隔，并在页面不可见时降低请求频率或暂停。

---

## 十六、代码质量与测试规范

### 16.1 基础质量要求

1. 提交代码前必须通过 lint 与类型检查。
2. 新增页面必须具备基础空态、错误态与 loading 态。
3. 复杂业务逻辑必须补充必要注释，说明输入、输出和边界。
4. 公共函数必须覆盖核心分支测试。

### 16.2 测试建议

以下场景建议优先补充测试：

1. 登录鉴权与路由守卫。
2. 列表查询条件与分页联动。
3. 任务状态轮询与失败重试。
4. 发布表单校验与提交流程。
5. 关键配置页保存与回显。

### 16.3 提测验收标准

页面交付前至少满足：

1. 无阻断性控制台报错。
2. 核心流程可从页面完整走通。
3. 接口异常分支可正确提示。
4. 主要页面在常见桌面分辨率下布局稳定。
5. 关键按钮权限控制正确。

## 十七、环境与配置规范

### 17.1 环境划分

前端至少区分以下环境：

1. 本地开发环境。
2. 测试环境。
3. 预发布环境。
4. 生产环境。

### 17.2 环境变量规范

1. 所有环境变量统一通过 Vite 约定方式管理。
2. 仅暴露前端运行所必需的公开配置。
3. 禁止在环境变量中放置服务端私密密钥。
4. API 地址、OSS 公网域名、埋点开关、调试开关需可配置。

---

## 十八、前后端协作规范

### 18.1 接口协作

1. 前后端在开发前必须对字段命名、状态值、分页格式、时间格式达成一致。
2. 所有枚举值都要有统一字典，不得在页面内各自解释。
3. 对长任务接口需明确创建接口、状态查询接口、结果接口三类能力。
4. 对发布流程需明确失败码、风控状态、授权失效状态的表现方式。

### 18.2 联调原则

1. 优先联调核心链路：登录 -> 新闻 -> 生成 -> 视频 -> 发布 -> 记录。
2. 接口未完成时可使用 Mock，但上线前必须切换为真实接口验证。
3. 对关键页面应保留接口示例与字段说明，便于后续维护。

## 十九、版本迭代规范

### 19.1 版本划分建议

1. `V1.0`：MVP后台，完成新闻、文案、视频、抖音发布、基础配置。
2. `V1.1`：补充快手发布、任务监控、日志查询。
3. `V1.2`：补充分析报表与权限体系完善。
4. `V2.0`：多平台、多语言、多内容领域扩展。

### 19.2 变更要求

1. 涉及路由、权限、接口结构、公共组件的修改必须同步更新文档。
2. 破坏性变更必须给出兼容方案或迁移说明。
3. UI 规范调整需同步更新设计基线与组件约定。

## 二十一、总结

本项目的前端开发应以“**后台可管理、任务可跟踪、结果可审核、发布可控制、数据可分析**”为核心目标展开。所有前端设计与实现都应围绕业务闭环、系统稳定性和后续扩展能力服务。

在执行层面，前端团队必须统一以下三点：

1. 统一技术栈：`Vue 3 + TypeScript + Vite + Element Plus + Vue Router + Pinia`。
2. 统一研发模式：业务分层、组件复用、接口类型化、状态可视化。
3. 统一质量标准：可维护、可扩展、可联调、可验收。

本规范作为当前版本前端实施基线，后续可随着项目迭代持续补充和完善。
