# EasyShorts AI 后端接口文档

> 适用于当前已完成的阶段一后端底座、新闻与内容链路，以及 Agent 智能处理模块。前端可以直接按本文档调用 `/easy-shorts` 下的接口。

## 1. 接口基础约定

### 1.1 Base URL

当前后端统一前缀为：

```text
/easy-shorts
```

本地开发环境示例：

```text
http://127.0.0.1:8000/easy-shorts
```

### 1.2 鉴权方式

需要登录的接口统一使用 JWT Bearer Token：

```http
Authorization: Bearer <access_token>
```

登录成功后，前端请保存 `access_token`，后续所有需要权限的接口都带上这个请求头。

说明：

- `GET /easy-shorts/health` 和 `POST /easy-shorts/auth/login` 不需要登录
- 其余阶段一接口默认都需要登录
- 超级管理员 `is_superuser=true` 会绕过权限码校验

### 1.3 统一响应格式

所有接口统一返回以下结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "f0b3d9f6c3b24d6e8a4d4d9e0b2d4a11"
}
```

字段说明：

- `code`：业务状态码，`0` 表示成功
- `message`：提示信息
- `data`：业务数据
- `request_id`：请求追踪 ID，排查问题时非常有用

常见错误返回示例：

```json
{
  "code": 40100,
  "message": "未认证或登录已失效",
  "data": {},
  "request_id": "f0b3d9f6c3b24d6e8a4d4d9e0b2d4a11"
}
```

### 1.4 常见错误码

| code | 含义 |
|---|---|
| 0 | 成功 |
| 40100 | 未认证或登录已失效 |
| 40300 | 无权限访问当前资源 |
| 40400 | 资源不存在 |
| 40900 | 资源状态冲突 |
| 42200 | 业务校验失败 |

### 1.5 分页约定

当前阶段只有部分列表接口使用分页参数：

- `GET /easy-shorts/system/users`
- `GET /easy-shorts/system/logs/operations`
- `GET /easy-shorts/system/logs/errors`
- `GET /easy-shorts/system/tasks`

分页请求参数：

```text
page=1
page_size=20
```

说明：

- `page` 从 `1` 开始
- `page_size` 默认 `20`
- 其他列表接口当前返回全量数据，不带分页参数

### 1.6 常用枚举

前端可按下面的枚举值做标签展示或表单校验：

- `UserStatus`：`ACTIVE`、`DISABLED`
- `MenuType`：`DIRECTORY`、`MENU`、`BUTTON`
- `TaskStatus`：`PENDING`、`RUNNING`、`SUCCESS`、`FAILED`、`RETRYING`、`CANCELLED`
- `PlatformAuthStatus`：`UNAUTHORIZED`、`AUTHORIZED`、`EXPIRED`、`DISABLED`
- `NewsSourceType`：`RSS`、`ATOM`、`WEB`、`MANUAL`
- `NewsFetchMode`：`MANUAL`、`SCHEDULED`
- `NewsStatus`：`NEW`、`DEDUPED`、`FILTERED`、`REJECTED`、`SCRIPT_READY`、`ARCHIVED`
- `AgentRunType`：`single_article`、`digest`、`push_plan`
- `AgentBizType`：`news`、`hot_topic`、`digest`
- `HotTopicStatus`：`ACTIVE`、`IGNORED`、`ARCHIVED`
- `PushPlanStatus`：`PENDING`、`SCHEDULED`、`EXECUTED`、`FAILED`、`CANCELLED`
- `PushPlanType`：`IMMEDIATE`、`DIGEST`、`SCHEDULED`
- `ConfigValueType`：`STRING`、`INTEGER`、`FLOAT`、`BOOLEAN`、`JSON`、`SECRET`
- `FileCategory`：`IMAGE`、`AUDIO`、`VIDEO`、`DOCUMENT`、`OTHER`
- `OperationStatus`：`SUCCESS`、`FAILED`

## 3. 接口详情

### 3.1 健康检查

**GET** `/easy-shorts/health`

不需要登录。

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "service": "EasyShorts AI Backend",
    "environment": "local",
    "api_prefix": "/easy-shorts"
  },
  "request_id": "f0b3d9f6c3b24d6e8a4d4d9e0b2d4a11"
}
```

---

### 3.2 登录认证

#### 3.2.1 用户登录

**POST** `/easy-shorts/auth/login`

不需要登录。

请求体：

```json
{
  "username": "admin",
  "password": "Admin@123456"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| username | string | 是 | 用户名，长度 3-64 |
| password | string | 是 | 密码，长度 6-128 |

响应体 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| access_token | string | JWT token |
| token_type | string | 固定为 `bearer` |
| expires_in | number | 过期秒数 |
| user | object | 当前登录用户信息 |

登录成功示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "access_token": "eyJhbGciOiJIUzI1NiJ9....",
    "token_type": "bearer",
    "expires_in": 43200,
    "user": {
      "id": 1,
      "username": "admin",
      "email": "admin@easyshorts.local",
      "phone": null,
      "nickname": "System Admin",
      "status": "ACTIVE",
      "is_superuser": true,
      "roles": [
        {
          "id": 1,
          "name": "超级管理员",
          "code": "super_admin",
          "description": "系统内置超级管理员",
          "is_system": true,
          "menu_ids": [1, 2, 3]
        }
      ],
      "permissions": [
        "system:user:list",
        "system:role:list"
      ],
      "last_login_at": "2026-04-14T10:00:00Z",
      "created_at": "2026-04-14T09:00:00Z",
      "updated_at": "2026-04-14T10:00:00Z"
    }
  },
  "request_id": "f0b3d9f6c3b24d6e8a4d4d9e0b2d4a11"
}
```

登录失败可能返回：

- `用户名或密码错误`
- `当前账号已禁用`

#### 3.2.2 退出登录

**POST** `/easy-shorts/auth/logout`

需要登录。

说明：

- 当前是 JWT 无状态模式
- 该接口主要用于后端记录退出日志
- 前端收到成功响应后，直接清理本地 token 即可

响应示例：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "success": true
  },
  "request_id": "f0b3d9f6c3b24d6e8a4d4d9e0b2d4a11"
}
```

#### 3.2.3 获取当前用户

**GET** `/easy-shorts/auth/me`

需要登录。

返回结构和登录成功里的 `user` 一致。

响应体 `data` 示例：

```json
{
  "id": 1,
  "username": "admin",
  "email": "admin@easyshorts.local",
  "phone": null,
  "nickname": "System Admin",
  "status": "ACTIVE",
  "is_superuser": true,
  "roles": [],
  "permissions": [],
  "last_login_at": "2026-04-14T10:00:00Z",
  "created_at": "2026-04-14T09:00:00Z",
  "updated_at": "2026-04-14T10:00:00Z"
}
```

### 3.3 用户管理

权限码：`system:user:list`

#### 3.3.1 用户列表

**GET** `/easy-shorts/system/users?page=1&page_size=20`

需要登录。

响应体 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| items | array | 用户列表 |
| page | number | 当前页 |
| page_size | number | 每页条数 |
| total | number | 总数 |

单个用户结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | number | 用户 ID |
| username | string | 用户名 |
| email | string \| null | 邮箱 |
| phone | string \| null | 手机号 |
| nickname | string \| null | 昵称 |
| status | string | `ACTIVE` / `DISABLED` |
| is_superuser | boolean | 是否超级管理员 |
| roles | array | 角色列表 |
| permissions | array[string] | 权限码列表 |
| last_login_at | string \| null | 最近登录时间 |
| created_at | string | 创建时间 |
| updated_at | string | 更新时间 |

#### 3.3.2 用户详情

**GET** `/easy-shorts/system/users/{user_id}`

需要登录。

响应体 `data` 和单个用户结构一致。

#### 3.3.3 新建用户

**POST** `/easy-shorts/system/users`

需要登录。

请求体：

```json
{
  "username": "demo",
  "password": "Demo@123456",
  "email": "demo@example.com",
  "phone": "13800000000",
  "nickname": "Demo User",
  "role_ids": [1, 2],
  "is_superuser": false,
  "status": "ACTIVE"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| username | string | 是 | 用户名，唯一 |
| password | string | 是 | 初始密码 |
| email | string \| null | 否 | 邮箱，唯一 |
| phone | string \| null | 否 | 手机号 |
| nickname | string \| null | 否 | 昵称 |
| role_ids | array[number] | 否 | 绑定的角色 ID 列表 |
| is_superuser | boolean | 否 | 是否超级管理员，默认 `false` |
| status | string | 否 | 默认 `ACTIVE` |

创建成功返回 `201`，`data` 为新用户对象。

#### 3.3.4 更新用户

**PUT** `/easy-shorts/system/users/{user_id}`

需要登录。

请求体：

```json
{
  "password": "NewPass@123",
  "email": "demo-new@example.com",
  "phone": "13900000000",
  "nickname": "Demo User New",
  "role_ids": [2],
  "is_superuser": false,
  "status": "DISABLED"
}
```

说明：

- `username` 不能改
- `role_ids` 传入后会覆盖当前角色关联
- `password` 传入后会重新加密

### 3.5 菜单管理

权限码：`system:menu:list`

#### 3.5.1 菜单树

**GET** `/easy-shorts/system/menus`

需要登录。

说明：

- 超级管理员会返回全部菜单
- 普通用户只返回自己有权限访问的菜单节点

返回结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "id": 1,
        "parent_id": null,
        "name": "dashboard",
        "title": "仪表盘",
        "path": "/dashboard",
        "component": "Dashboard",
        "icon": "dashboard",
        "permission_code": null,
        "menu_type": "MENU",
        "sort_order": 1,
        "hidden": false,
        "created_at": "2026-04-14T09:00:00Z",
        "updated_at": "2026-04-14T09:00:00Z",
        "children": []
      }
    ]
  },
  "request_id": "..."
}
```

说明：

- 返回的是树结构
- 每个节点都带 `children`
- `menu_type` 取值：`DIRECTORY`、`MENU`、`BUTTON`

#### 3.5.2 新建菜单

**POST** `/easy-shorts/system/menus`

需要登录。

请求体：

```json
{
  "parent_id": null,
  "name": "user_mgmt",
  "title": "用户管理",
  "path": "/system/users",
  "component": "SystemUsers",
  "icon": "users",
  "permission_code": "system:user:list",
  "menu_type": "MENU",
  "sort_order": 10,
  "hidden": false
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| parent_id | number \| null | 否 | 父级菜单 ID |
| name | string | 是 | 菜单标识，同级唯一 |
| title | string | 是 | 菜单标题 |
| path | string \| null | 否 | 路由路径 |
| component | string \| null | 否 | 前端组件名 |
| icon | string \| null | 否 | 图标 |
| permission_code | string \| null | 否 | 权限码 |
| menu_type | string | 否 | 默认 `MENU` |
| sort_order | number | 否 | 排序，默认 `0` |
| hidden | boolean | 否 | 是否隐藏 |

### 3.12 新闻与内容

这部分是阶段二新增的主链路接口，覆盖新闻源管理、内容采集、新闻列表、新闻详情、抓取记录和内容生成。

当前采集能力说明：

- `RSS` / `ATOM`：直接抓取 feed
- `WEB`：抓取网页列表页或落地页并抽取正文，支持配置候选文章地址
- `MANUAL`：从 `extra.items` / `extra.entries` 导入结构化数据
- 定时采集只轮询 `RSS`、`ATOM`、`WEB`，`MANUAL` 只支持手动同步

#### 3.12.1 新闻源管理

权限码：

- 列表/详情：`news:source:list`
- 新建：`news:source:create`
- 更新：`news:source:update`
- 同步：`news:source:sync`

接口清单：

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/easy-shorts/news/sources` | 新闻源列表 |
| GET | `/easy-shorts/news/sources/{source_id}` | 新闻源详情 |
| POST | `/easy-shorts/news/sources` | 新建新闻源 |
| PUT | `/easy-shorts/news/sources/{source_id}` | 更新新闻源 |
| POST | `/easy-shorts/news/sources/{source_id}/sync` | 手动同步新闻源 |

新闻源列表返回的核心字段：

| 字段 | 说明 |
|---|---|
| `source_key` | 新闻源标识，例如 `openai_blog` |
| `name` | 新闻源名称 |
| `source_type` | 来源类型，`RSS` / `ATOM` / `WEB` / `MANUAL` |
| `url` | RSS / Atom 地址，或者 WEB 落地页地址 |
| `category` | 业务分类 |
| `language` | 语言标识 |
| `fetch_interval_minutes` | 定时抓取间隔 |
| `is_enabled` | 是否启用 |
| `last_fetched_at` | 最近一次抓取时间 |
| `last_success_at` | 最近一次成功时间 |
| `last_error_message` | 最近一次错误信息 |
| `extra` | 扩展配置，JSON 对象 |

新闻源保存规则：

- `source_key` 会在后端自动转成小写，并且全局唯一
- `url` 也必须唯一，重复会直接返回冲突错误
- `source_type` 决定采集策略，前端创建时要和 `url`、`extra` 一起填对

新建/更新请求示例：

```json
{
  "source_key": "openai_blog",
  "name": "OpenAI Blog",
  "source_type": "RSS",
  "url": "https://openai.com/blog/rss.xml",
  "category": "模型发布",
  "language": "en",
  "fetch_interval_minutes": 360,
  "is_enabled": true,
  "extra": {
    "weight": 25
  }
}
```

`extra` 的常用约定：

| 来源类型 | 常用字段 | 说明 |
|---|---|---|
| `RSS` / `ATOM` | `weight` | 可选热度权重，参与热度评分 |
| `WEB` | `mode` | `list` 或 `single`，默认 `list` |
| `WEB` | `article_urls` / `urls` | 文章页地址列表，优先直接抓这些地址 |
| `WEB` | `max_items` | 单次最多抓取数量，默认 `10` |
| `WEB` | `link_patterns` | 文章地址或锚文本的正则白名单 |
| `WEB` | `exclude_link_patterns` | 文章地址或锚文本的正则黑名单 |
| `MANUAL` | `items` / `entries` | 手工导入的数据数组 |

`MANUAL` 每条结构化数据建议包含这些字段：

| 字段 | 说明 |
|---|---|
| `title` | 标题 |
| `link` / `url` | 原始地址 |
| `content` / `description` / `summary` | 正文或摘要 |
| `published_at` / `published_time` / `pubDate` / `date` | 发布时间 |
| `author` / `creator` | 作者 |
| `guid` / `id` | 唯一标识 |

手动同步接口会返回 `202 Accepted`，`data` 里是任务信息。前端可以：

1. 直接去任务中心看任务状态
2. 等 worker 执行完后刷新新闻列表或抓取记录页

同步任务完成后，任务结果里会返回一组采集汇总信息，典型结构如下：

```json
{
  "source": {
    "id": 1,
    "source_key": "openai_blog",
    "name": "OpenAI Blog",
    "source_type": "RSS"
  },
  "fetch_record": {
    "id": 1,
    "status": "SUCCESS"
  },
  "items": [],
  "total_count": 12,
  "new_count": 8,
  "duplicate_count": 2,
  "filtered_count": 6,
  "rejected_count": 2
}
```

说明：

- `items` 是本次新入库或更新后的新闻摘要列表
- `duplicate_count` 表示去重掉的条数
- `filtered_count` 表示通过筛选并生成了摘要 / 脚本的条数
- `rejected_count` 表示被敏感词或热度规则拒绝的条数
- `WEB` 源也会被定时调度，`MANUAL` 源不会进入定时轮询

新闻列表常用过滤参数：

| 参数 | 类型 | 说明 |

新闻列表项常用字段：

| 字段 | 说明 |
|---|---|
| `id` | 新闻 ID |
| `title` | 新闻标题 |
| `content` | 原始正文，列表页只返回截断内容 |
| `source` | 来源名称 |

| `source_url` | 来源地址 |
| `url` | 原始新闻地址 |

| `language` | 语言 |

- `source_detail`：新闻源完整信息
- `fetch_records`：当前新闻源最近的抓取记录

新闻状态建议按下面的链路理解：

- `NEW`：刚入库
- `DEDUPED`：完成基础去重
- `FILTERED`：通过筛选并完成摘要 / 标题 / 脚本生成
- `REJECTED`：被敏感词或热度规则拒绝
- `SCRIPT_READY`：已完成二次生成
- `ARCHIVED`：已归档

内容生成接口请求示例：

```json
{
  "style": "professional",
  "regenerate": true
}
```

内容处理接口请求示例：

```json
{
  "style": "professional",
  "force": false
}
```

说明：

- 内容处理会重新跑正文清洗、去重、翻译、摘要、标签分类和过滤
- 内容生成是异步任务，接口同样返回 `202 Accepted`
- 生成完成后，新闻状态会更新为 `SCRIPT_READY`
- 如果新闻处于 `REJECTED`，默认不允许直接生成，除非后续再明确放开策略
- 生成结果会回写到新闻表，通常包括 `summary`、`translated_title`、`translated_content`、`script`、`tags` 和 `raw_metadata.generation`

- 后端定时任务会自动扫描 `is_enabled=true` 且类型为 `RSS`、`ATOM`、`WEB` 的新闻源
- 如果 `last_fetched_at` 早于当前间隔，系统会把该来源加入待采集队列
- `MANUAL` 类型不参与定时扫描，只能通过手动同步接口触发
- 前端在列表页可以重点展示 `is_enabled`、`last_fetched_at`、`last_success_at`、`last_error_message` 这几个字段，方便运营排查

### 3.13 Agent 智能处理模块

这部分接口对应后端的智能处理链路，覆盖模型配置、单条新闻智能分析、热点池、简报、推送计划和推送记录。

当前模型配置说明：

- 默认模型：`qwen3.5-plus`
- 可选模型：来自 `agent.supported_models`
- 当前默认提供方：`dashscope`
- 如果配置了 `DASHSCOPE_API_KEY`，后端会优先走 AgentScope 的 DashScope 模型；没有 key 时会自动回退到规则化实现
- 配置入口：`GET /easy-shorts/agent/config` 和 `PUT /easy-shorts/agent/config`

权限码：

- 配置列表：`agent:config:list`
- 配置更新：`agent:config:update`
- 模型列表：`agent:model:list`
- 触发运行：`agent:run:create`
- 简报运行：`agent:run:digest:create`
- 运行列表：`agent:run:list`
- 运行重试：`agent:run:retry`
- 步骤重试：`agent:run:step:retry`
- 热点重处理：`agent:hot-topic:reprocess`
- 热点列表：`agent:hot-topic:list`
- 简报列表：`agent:digest:list`
- 简报推送：`agent:digest:push`
- 推送计划列表：`agent:push-plan:list`
- 推送计划执行：`agent:push-plan:execute`
- 推送记录列表：`agent:push-record:list`

## 5. 前端接入建议

1. 登录后把 `access_token` 存到本地缓存或状态管理里
2. 统一在请求拦截器里加上 `Authorization: Bearer <token>`
3. 遇到 `40100` 时跳转登录页
4. 遇到 `40300` 时给出无权限提示
5. 所有接口都保留 `request_id`，便于后端排障
6. 上传文件使用 `multipart/form-data`，不要当普通 JSON 发
