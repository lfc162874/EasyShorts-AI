## 一、文档目的

本文档专门用于定义 `AI 热点监控助手` 中“采集”这一条链路的开发方案。  
这里的“采集”不是单点爬虫功能，而是一套完整的业务能力，覆盖：

1. 内容来源管理
2. 抓取调度与执行
3. 正文解析与去重
4. 采集结果查看
5. 抓取记录追踪
6. 异常重试与日志排查

本文档同时面向前端和后端：

- 前端可以直接按本文档设计页面、字段、按钮和交互；
- 后端可以直接按本文档设计表结构、接口、任务流和状态流转；
- 联调时双方可以以本文件作为统一语义基线。

## 二、与当前代码的命名映射

由于当前代码基于早期“新闻采集”方向实现，因此代码里仍沿用 `news` 命名。  
在 `v3` 的“AI 热点监控助手”方向下，建议按下面方式理解：

| 当前代码命名 | V3 业务语义 | 说明 |
| --- | --- | --- |
| `news_sources` | 内容来源 | 监控源、站点源、RSS 源 |
| `news` | 采集内容 / 文章内容 | 被抓取并解析后的文章 |
| `news_fetch_records` | 采集记录 | 一次采集任务的执行记录 |
| `/news/sources` | 来源管理接口 | 后续可平滑重命名为 `/sources` |
| `/news/records` | 采集记录接口 | 后续可平滑重命名为 `/crawl-records` |
| `/news` | 内容中心接口 | 后续可平滑重命名为 `/articles` |

结论是：**当前代码可以继续用，不需要立刻重命名数据库和模块，只要在前端和文档层面统一成“来源 / 采集 / 内容”语义即可。**

## 三、采集模块的目标

采集模块要解决的不是“把网页抓下来”，而是以下业务问题：

1. 系统要监控哪些 AI 网站、RSS、博客或资讯源；
2. 每个来源如何抓、多久抓一次；
3. 每次抓取是否成功，抓回来了多少内容；
4. 抓回来的内容是否重复、是否可读、是否需要过滤；
5. 运维人员如何查看结果、排查错误、重新执行；
6. 分析模块如何消费这些结构化结果。

因此，采集模块必须同时具备：

- 管理入口
- 执行入口
- 结果入口
- 记录入口
- 排障入口

## 4.2 来源管理页

**建议路由：**

- 当前实现可继续使用：`/news/sources`
- V3 推荐命名：`/sources`

### 页面定位

这是采集模块的核心入口，负责管理“系统监控哪些来源”。

### 列表字段建议

| 字段 | 说明 |
| --- | --- |
| 来源名称 | 例如 OpenAI Blog、Anthropic News |
| 来源类型 | RSS / ATOM / WEB / MANUAL |
| 来源分类 | 官方博客 / 媒体 / 学术 / 社区 |
| 来源地址 | 来源 URL |
| 语言 | `zh` / `en` |
| 抓取频率 | 例如 10 分钟、60 分钟、360 分钟 |
| 状态 | 启用 / 停用 |
| 上次抓取时间 | 最近一次尝试抓取时间 |
| 上次成功时间 | 最近一次成功抓取时间 |
| 最近错误 | 最近失败原因摘要 |
| 操作 | 编辑、启停、立即抓取、查看记录、查看内容 |

### 页面支持操作

1. 新增来源
2. 编辑来源
3. 启用 / 停用来源
4. 立即抓取
5. 查看来源详情
6. 跳转抓取记录
7. 跳转采集内容

### 新增 / 编辑表单字段建议

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| 来源名称 | 是 | 页面显示名称 |
| 来源标识 `source_key` | 是 | 唯一键，便于系统识别 |
| 来源类型 | 是 | `RSS` / `ATOM` / `WEB` / `MANUAL` |
| 来源 URL | 是 | 采集入口地址 |
| 来源分类 | 否 | 如官方、媒体、学术、社区 |
| 语言 | 是 | 默认 `en` 或 `zh` |
| 抓取频率 | 是 | 单位分钟 |
| 是否启用 | 是 | 是否参与定时采集 |
| 扩展配置 `extra` | 否 | 存放权重、规则、定制参数 |

### `WEB` 类型建议扩展字段

如果后续需要提升网页采集灵活性，建议 `extra` 中支持：

- 列表页选择器
- 标题选择器
- 正文选择器
- 发布时间选择器
- 摘要选择器
- 链接选择器
- 自定义请求头

V1 阶段可以先不暴露成高级可视化配置页，只保留 JSON 扩展字段即可。

## 五、采集模块的后端设计

## 5.1 后端职责划分

采集模块在后端建议拆成以下层次：

1. API 层：接收页面请求、返回统一结构
2. Service 层：实现来源管理、采集、去重、解析逻辑
3. Task 层：执行异步抓取和批量调度
4. Model 层：定义来源、内容、抓取记录、任务实体
5. Integration 层：封装 RSS / WEB 等来源适配方式

### 当前代码中的对应模块

| 责任 | 当前文件 |
| --- | --- |
| API | `backend/app/api/v1/news.py` |
| Service | `backend/app/services/news_service.py` |
| Task | `backend/app/tasks/news_tasks.py` |
| 调度派发 | `backend/app/services/task_service.py` |
| 数据模型 | `backend/app/db/models/business.py` |

## 5.4 采集任务链路设计

### 手动抓取链路

```text
前端点击“立即抓取”
        │
        ▼
POST /easy-shorts/news/sources/{source_id}/sync
        │
        ▼
创建 TaskJob
        │
        ▼
派发 Celery 任务 sync_news_source_task
        │
        ▼
news_service.sync_news_source 执行抓取
        │
        ▼
解析、去重、入库、记录统计
        │
        ▼
更新 news_fetch_records 和 task_jobs
```

### 定时抓取链路

```text
Celery Beat 定时触发
        │
        ▼
sync_due_news_sources_task
        │
        ▼
找出到期且启用的来源
        │
        ▼
逐个执行 sync_news_source
        │
        ▼
写入采集结果与抓取记录
```

### 去重链路

当前建议采用以下顺序：

1. 优先按 `url` 去重
2. 再按 `dedup_hash` 去重
3. 后续如噪音较高，可增加标题相似度去重

## 5.5 采集服务设计建议

从实现角度，采集服务建议保持“确定性逻辑在服务层，智能判断在后续分析层”。

### 采集服务需要做的事

1. 拉取远端内容
2. 解析标题、正文、摘要、时间、作者等基础信息
3. 做 URL 去重和哈希去重
4. 写入数据库
5. 记录抓取统计
6. 记录失败原因

### 采集服务暂时不应该做太重的事

1. 不做复杂大模型改写
2. 不做高成本智能排序
3. 不做视频或媒体处理
4. 不在采集阶段塞太多业务判断

采集阶段的目标应该是：**稳定拿到结构化内容**。

2. 动态页面抓取
3. 可视化规则调试

## 八、采集模块验收标准

采集模块完成后，至少应满足以下条件：

1. 可以新增一个 RSS 或网页来源；
2. 可以手动触发一次抓取；
3. 抓取结果能够进入内容中心；
4. 重复内容不会重复入库；
5. 抓取失败能够在抓取记录页看到错误原因；
6. 定时任务能够按频率执行；
7. 前端可以从来源页跳转到抓取记录页和内容页；
8. 后端保留完整的抓取记录和任务状态；
9. 当前语义虽然仍沿用 `news` 命名，但业务上已可支撑 `v3` 的“来源 - 采集 - 内容”闭环。

---

## 九、下一步建议

基于当前代码和 `v3` 方向，我建议下一步这样推进：

1. 前端先把“新闻源管理 / 抓取记录 / 新闻管理”页面文案改成 `v3` 语义；
2. 后端继续复用现有 `/news/...` 接口，不做立刻重构；
3. 在采集模块稳定后，再新增“热点分析”和“自动推送”模块；
4. 等分析、推送主链路跑顺后，再考虑做统一命名迁移。

---

## 十、结论

采集模块现在最重要的不是继续扩展花哨能力，而是把以下三件事做好：

1. **来源可管理**
2. **抓取可追踪**
3. **内容可消费**

只要这三件事前后端都打通，采集模块就已经足够支撑 `AI 热点监控助手 v3` 的第一阶段开发。
