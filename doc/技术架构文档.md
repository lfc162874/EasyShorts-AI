# 一、文档概述

## 1.1 编写目的

本文档用于描述 AI 热点监控助手的整体技术架构设计，包括系统分层、核心模块、技术选型、数据架构及部署方案，为系统开发、部署和维护提供技术依据。

## 1.2 项目背景

AI 行业信息更新极快，OpenAI、Anthropic、Google DeepMind、Hugging Face、arXiv、TechCrunch、36Kr 等站点每天都会产生大量内容。  
本项目的核心目标不是“搬运信息”，而是自动发现值得关注的热点，并把结果以摘要、解读和推送的方式交给用户。

## 1.3 项目目标

* 自动采集多个 AI 网站与 RSS 资讯源。
* 自动识别和分析热点话题。
* 利用智能体完成摘要、解读与优先级判断。
* 根据订阅规则自动推送热点结果。
* 构建可扩展的 AI 资讯监控平台。

## 1.4 文档范围

本技术架构文档涵盖：

* 系统总体架构设计
* 核心模块设计
* 技术选型说明

# 三、系统分层设计

## 3.1 表现层（Frontend）

**技术栈：Vue 3 + Vite + TypeScript + Element Plus**

### 职责

* 系统管理与配置
* 来源管理与订阅管理
* 热点列表与详情查看
* 推送记录与任务监控
* 操作日志、错误日志、任务日志查看

### 核心页面

* 内容来源管理
* 热点内容列表
* 热点详情与摘要
* 订阅规则配置
* 推送记录查询
* 系统配置中心

## 3.2 接口层（API Layer）

**技术栈：FastAPI**

### 职责

* 提供 RESTful API 接口
* 用户认证与授权
* 调度 Agent 执行任务
* 返回系统状态、任务结果与推送结果

### 核心接口

* 内容来源管理接口
* 热点内容接口
* 订阅规则接口
* 推送记录接口
* 系统配置接口

## 3.3 Agent 编排层（Agent Orchestration Layer）

**技术栈：AgentScope**

Agent 在这里主要承担“分析与决策”的角色，不直接耦合底层抓取实现。  
采集、解析、去重、入库等确定性逻辑优先放在服务层与任务层，Agent 负责对结果做更高层的判断与总结。

| Agent 名称 | 职责 |
| --- | --- |
| SourceMonitorAgent | 监控来源状态与调度优先级 |
| CollectorAgent | 选择采集适配器并拉取内容 |
| NormalizerAgent | 规范化标题、正文、时间与来源信息 |
| HotspotAnalysisAgent | 进行热度评分、聚类与趋势判断 |
| SummaryAgent | 生成摘要、要点与影响分析 |
| PushDecisionAgent | 决定是否即时推送及推送渠道 |

## 3.4 数据采集与解析层（Data Acquisition & Parsing Layer）

这是本项目最关键的基础能力之一。  
当前阶段的最佳方案不是一上来就使用重型爬虫系统，而是采用“**轻量采集适配器 + 异步任务调度**”的方式，把 RSS、网页正文和动态页面拆开处理。

### 推荐技术选型

| 场景 | 技术 | 作用 |
| --- | --- | --- |
| RSS / Atom | `feedparser` | 解析 RSS 与 Atom，简单稳定 |
| 普通网页抓取 | `httpx` 或 `requests` | 发起 HTTP 请求，获取页面内容 |
| HTML 解析 | `BeautifulSoup4 + lxml` | 解析 DOM 与正文结构 |
| 正文抽取 | `readability-lxml` 或 `trafilatura` | 提取文章正文、标题、摘要 |
| 动态页面 | `Playwright` | 处理 JS 渲染页面和懒加载内容 |
| 定时调度 | `Celery Beat` | 周期性触发来源抓取任务 |
| 异步执行 | `Celery Worker` | 执行抓取、解析、去重、分析任务 |
| 队列与缓存 | `Redis` | 任务队列、短期去重、限流与缓存 |

### 为什么不把 Scrapy 作为首选主框架

`Scrapy` 很适合大规模网页爬取，但本项目当前阶段的核心来源以 RSS、官方博客、少量稳定资讯站点为主，更需要的是：

1. 低依赖、易维护。
2. 对 RSS/网页/动态页都能按需适配。
3. 和 Celery 的异步任务体系天然协作。
4. 便于后续把采集结果接到 Agent 分析链路中。

因此，当前更适合采用：

* `feedparser + httpx/requests + BeautifulSoup4/lxml` 作为主栈。
* `Playwright` 作为动态站点兜底方案。
* `Scrapy` 作为后期来源规模扩大后的可选增强，不作为首版主框架。

### 采集流程

```text
来源配置
   │
   ▼
采集适配器选择（RSS / Web / Dynamic）
   │
   ▼
抓取原始内容
   │
   ▼
正文抽取与结构化解析
   │
   ▼
去重与质量过滤
   │
   ▼
写入 news / news_fetch_records
```

## 3.5 任务执行层（Task Execution Layer）

**技术栈：Celery + Redis**

### 职责

* 执行耗时任务
* 实现异步处理
* 支持任务重试
* 承接定时抓取、热点分析、摘要生成、推送执行

### 核心任务

* 来源采集任务
* 内容解析任务
* 热点识别任务
* 摘要与解读任务
* 推送任务

## 3.6 数据与存储层（Data Layer）

| 组件 | 技术 | 作用 |
| --- | --- | --- |
| 关系数据库 | PostgreSQL | 存储系统配置、内容、订阅与任务数据 |
| 缓存与队列 | Redis | 缓存、任务队列、限流与短期去重 |

# 五、技术选型说明

| 模块 | 技术 | 选择理由 |
| --- | --- | --- |
| 前端 | Vue 3 + Vite + TypeScript + Element Plus | 生态成熟，适合后台系统 |
| 后端 | FastAPI | 高性能、支持异步、接口表达清晰 |
| Agent 框架 | AgentScope | 支持多智能体协作与任务编排 |
| 数据采集 | `feedparser` + `httpx/requests` + `BeautifulSoup4/lxml` + `trafilatura/readability-lxml` + `Playwright` | 兼顾 RSS、静态网页和动态页面 |
| 异步任务 | Celery + Redis | 支持分布式任务调度与重试 |
| 数据库 | PostgreSQL | 稳定可靠，支持 JSON 与复杂查询 |
| 缓存 | Redis | 适合缓存、队列、限流和短期去重 |
| 对象存储 | 阿里云 OSS | 适合扩展文件与媒资存储 |
| 定时任务 | Celery Beat | 适合周期性采集与汇总任务 |
| 容器化 | Docker | 易部署、易迁移、易扩展 |
| 反向代理 | Nginx | 提供负载均衡与静态资源分发 |

> 说明：如果未来来源数量和反爬复杂度显著提升，可以引入 `Scrapy` 作为增强型采集框架，但当前版本不建议把它作为唯一主框架。

# 十二、项目亮点（用于简历）

* 面向 AI 行业热点监控的垂直智能体系统，聚焦“采集、分析、推送”的完整闭环。
* 采用轻量采集适配器方案，兼顾 RSS、网页与动态页面场景。
* 结合 AgentScope、FastAPI 与 Celery 构建异步、可追踪、可扩展的任务体系。
* 使用 PostgreSQL、Redis 与 OSS 完成结构化数据与扩展资源管理。
* 支持订阅、摘要、热点识别与自动推送，具备持续演进空间。
