## 3. 后端总体原则

### 3.1 基本原则

- 单一职责：路由层、应用服务层、领域模型层、任务层、基础设施层职责分离
- 明确边界：同步请求只处理轻逻辑，耗时任务必须异步化
- 幂等优先：采集、生成、发布、回调等动作必须具备幂等能力
- 失败可恢复：外部依赖调用必须支持重试、补偿与人工介入
- 数据可追踪：核心业务链路必须能从新闻追溯到脚本、视频、发布记录
- 安全默认开启：密钥、账号、发布动作、内容审核按高风险流程治理
- 可扩展优先：平台接入、模型接入、内容源接入必须采用插件化/适配器思路

### 3.2 分层原则

后端按以下逻辑分层：

1. API Layer：接收请求、参数校验、返回统一响应
2. Application Layer：编排业务流程、事务控制、状态流转
3. Domain Layer：定义实体、值对象、状态枚举、业务规则
4. Task Layer：处理异步任务、重试、超时、补偿
5. Integration Layer：封装 LLM、RSS、OSS、TTS、平台发布等外部依赖
6. Infrastructure Layer：数据库、缓存、日志、配置、监控、消息队列

禁止出现以下情况：

- 在路由层直接写复杂业务逻辑
- 在路由层直接操作 ORM Session 多步提交
- 在 Celery 任务中直接拼接 SQL 或散落业务规则
- 在业务代码中硬编码平台密钥、模型参数、Prompt 模板
- 由前端拼装关键业务状态并作为可信输入写回数据库

## 4. 模块职责规范

### 4.1 核心业务模块

根据需求文档，后端至少拆分为以下模块：

| 模块 | 对应需求 | 职责 |
| --- | --- | --- |
| `news` | FR-001 ~ FR-005 | 新闻抓取、解析、去重、入库、来源管理 |
| `content` | FR-006 ~ FR-015 | 热点筛选、分类、翻译、摘要、脚本生成、标题标签生成 |
| `media` | FR-016 ~ FR-025 | 图片、配音、字幕、视频规划、视频合成 |
| `publish` | FR-026 ~ FR-030 | 平台鉴权、发布、回调、发布记录、定时发布 |
| `analytics` | FR-031 ~ FR-034 | 播放/点赞/评论统计、趋势分析、报表聚合 |
| `system` | FR-035 ~ FR-039 | 用户、配置、日志、调度、数据源管理 |

### 4.2 模块交互约束

- `news` 模块不得直接依赖 `publish` 模块
- `publish` 模块只消费已审核且状态合规的视频资产
- `analytics` 模块仅做统计与聚合，不反向修改生产主状态
- `system` 模块只负责配置和管理，不承载核心业务编排
- 模块之间优先通过应用服务层交互，不允许跨模块直接读写内部实现细节

## 6. 编码规范

### 6.1 通用规范

- Python 版本建议统一为 3.11+
- 代码风格遵循 PEP 8
- 类型标注必须完整，公共函数禁止省略关键类型
- 所有公共方法必须具备简洁且准确的 docstring
- 枚举值、状态机常量统一集中定义，禁止魔法字符串散落
- 时间字段统一使用 UTC 存储，展示时由前端按时区转换

### 6.2 命名规范

- 包、模块、文件名：`snake_case`
- 类名：`PascalCase`
- 函数、变量：`snake_case`
- 常量：`UPPER_SNAKE_CASE`
- 数据库表名：复数或业务自然名统一即可，但全项目必须一致；建议采用 `snake_case`
- API 路径：名词化、复数化，并统一挂载业务标识 `easy-shorts`，例如 `/easy-shorts/news`

### 6.3 代码分层规范

- 路由函数只做四件事：鉴权、校验、调用服务、返回结果
- 服务层函数必须体现业务语义，如 `create_video_from_news`、`submit_publish_job`
- Repository 接口命名应明确数据语义，如 `get_by_id`、`find_by_url_hash`、`save`
- 外部集成类统一以 `Client` 或 `Gateway` 结尾，如 `DouyinPublisherClient`
- Celery 任务函数统一以 `_task` 结尾

## 7. API 开发规范

### 7.1 协议规范

- API 风格：RESTful
- 统一路径前缀：`/easy-shorts`
- 数据格式：`application/json`
- 字符编码：UTF-8
- 对外响应必须为统一结构

说明：

- `easy-shorts` 作为本项目后端统一接口标识，所有业务接口默认在该前缀下暴露
- 当前接口路径不再保留 `/api/v1` 版本段
- 推荐接口示例：
  - 新闻管理：`/easy-shorts/news`
  - 视频管理：`/easy-shorts/videos`
  - 发布管理：`/easy-shorts/publish-records`
  - 系统配置：`/easy-shorts/system/configs`

推荐响应结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "a3d6d4d9b6f44dc2a93587f8d2e33987"
}
```

分页结构推荐：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [],
    "page": 1,
    "page_size": 20,
    "total": 100
  },
  "request_id": "..."
}
```

### 7.2 状态码规范

- `200`：查询成功、更新成功
- `201`：创建成功
- `202`：异步任务已受理
- `400`：参数错误
- `401`：未认证
- `403`：无权限
- `404`：资源不存在
- `409`：状态冲突或幂等冲突
- `422`：业务校验失败
- `429`：限流
- `500`：系统内部错误
- `502/503`：外部依赖异常或服务不可用

### 7.3 入参与出参规范

- 使用 Pydantic 定义 Request/Response Schema
- 禁止直接暴露 ORM 模型作为 API 出参
- 所有写接口必须校验字段合法性、长度、枚举值、敏感词和状态前置条件
- 所有列表接口必须支持分页，禁止默认返回无限结果集
- 所有查询接口支持明确排序字段，避免隐式顺序

### 7.4 幂等规范

以下接口必须具备幂等能力：

- 新闻抓取入库
- 视频生成任务提交
- 发布任务提交
- 平台回调处理
- 定时任务重复触发

推荐方式：

- 业务唯一键约束
- 幂等键 `idempotency_key`
- 去重指纹，如 URL 哈希、内容摘要哈希、任务签名

## 8. 业务状态机规范

### 8.1 新闻状态

建议统一使用以下状态：

- `NEW`：新抓取
- `DEDUPED`：已去重
- `FILTERED`：已完成筛选
- `REJECTED`：被过滤
- `SCRIPT_READY`：脚本可生成
- `ARCHIVED`：归档

### 8.2 视频状态

- `INIT`
- `SCRIPT_GENERATED`
- `AUDIO_READY`
- `SUBTITLE_READY`
- `RENDERING`
- `RENDER_SUCCESS`
- `RENDER_FAILED`
- `COMPLIANCE_PENDING`
- `COMPLIANCE_REJECTED`
- `PUBLISH_READY`
- `PUBLISHED`

### 8.3 发布状态

- `PENDING`
- `SCHEDULED`
- `SUBMITTING`
- `SUCCESS`
- `FAILED`
- `RETRYING`
- `CANCELLED`

状态流转要求：

- 只能按预定义状态图流转，禁止任意跳转
- 状态变更必须落库并记录操作时间、操作者、原因
- 失败状态必须带失败码与失败原因
- 手工干预必须可审计

## 9. 数据库开发规范

### 9.1 基本要求

- PostgreSQL 为唯一业务事实源
- 所有表必须包含主键、创建时间、更新时间
- 审计敏感表建议包含创建人、更新人
- 所有外键必须明确约束或明确说明为何不建外键
- 所有状态字段必须使用受控枚举
- 优先使用 `JSONB` 存储半结构化配置、Agent输出原文、第三方回包

### 9.2 核心表建议

除需求文档中的 `news`、`video`、`publish_record` 外，建议补充以下表：

- `news_source`：新闻来源配置
- `task_execution`：异步任务执行记录
- `platform_account`：平台账号与授权信息
- `content_audit_record`：内容审核记录
- `system_config`：系统配置中心
- `operation_log`：关键操作日志

### 9.3 字段规范

- 主键统一使用 `BIGSERIAL` 或雪花 ID，项目内保持一致
- 时间字段使用 `TIMESTAMP WITH TIME ZONE` 或统一 UTC 规范
- 布尔字段使用 `BOOLEAN`
- 金额、评分等字段禁止使用浮点型时需改用 `NUMERIC`
- URL、对象存储地址、外部任务 ID 必须保留足够长度

### 9.4 索引规范

必须为以下场景建立索引：

- 新闻唯一性去重：`url`、`url_hash`
- 任务查询：`status`、`created_at`
- 发布记录追踪：`video_id`、`platform`、`status`
- 定时调度：`schedule_time`、`status`
- 审核查询：`compliance_status`

### 9.5 迁移规范

- 数据库变更统一通过 Alembic 管理
- 禁止手工改生产表结构而不补迁移脚本
- 每个迁移脚本必须说明变更目的与回滚风险
- 破坏性变更必须采用灰度方案：先兼容、后切流、再清理

## 10. Repository 与事务规范

### 10.1 Repository 规范

- 一个 Repository 对应一个聚合根或核心实体
- 不在 Repository 中拼业务状态机
- 查询方法应显式说明过滤条件和锁策略
- 对高并发状态更新场景，必须明确使用乐观锁或悲观锁

### 10.2 事务规范

- 事务边界放在 Service 层
- 单次事务只覆盖必须原子化的数据库写操作
- 禁止在数据库事务中执行耗时外部调用
- 外部调用成功后需要回写数据库时，必须考虑补偿逻辑
- 发布动作、上传动作、回调动作必须设计防重和补偿

## 11. 异步任务规范

### 11.1 适用范围

以下操作必须异步执行：

- 新闻批量抓取
- LLM 摘要、翻译、脚本生成
- 配音生成
- 字幕对齐
- 视频渲染
- 平台发布
- 数据统计回收

### 11.2 Celery 任务要求

- 每个任务必须设置超时时间
- 每个任务必须配置最大重试次数
- 每个任务必须记录输入摘要、执行耗时、最终状态
- 每个任务必须支持链路追踪字段透传，如 `request_id`、`task_id`
- 同一任务重复执行时必须保证幂等

### 11.3 队列划分建议

- `news_queue`：抓取与解析
- `content_queue`：LLM 生成与翻译
- `media_queue`：音频、字幕、视频处理
- `publish_queue`：平台发布
- `analytics_queue`：统计回收与报表

### 11.4 重试与补偿

- 网络抖动、平台限流、瞬时依赖异常可自动重试
- 内容违规、参数错误、资源缺失属于业务失败，不得无限重试
- 连续失败超过阈值时，必须进入人工处理队列
- 已上传但未落库、已发布但回写失败等场景必须设计补偿任务

## 12. Agent 与 AI 能力接入规范

### 12.1 Agent 职责边界

- Agent 负责决策、生成、筛选、规划
- 业务服务负责状态管理、事务控制、持久化
- 不允许由 Agent 直接改数据库最终状态
- Agent 输出必须经过结构化校验后才能进入正式流程

### 12.2 Prompt 与模型管理

- Prompt 模板统一配置化管理，不得散落在业务代码
- 模型名称、温度、token 限额、超时配置统一集中管理
- 对生成结果必须增加兜底校验：长度、敏感词、结构完整性、平台规范
- 对重要生成产物必须保留原始输入、原始输出、最终入库结果

### 12.3 AI 输出落地要求

- 摘要、标题、脚本、标签必须保留版本信息
- 需要支持重新生成，并与历史版本可追溯
- 关键 AI 决策结果要可解释，至少保留评分项与筛选原因

## 13. 外部集成规范

### 13.1 集成方式

所有第三方依赖必须通过 `integrations` 层封装，包括：

- RSS/网页抓取源
- LLM 服务
- TTS 服务
- OSS 存储
- 抖音/快手/B站/小红书等发布平台

### 13.2 接入要求

- 每个外部系统必须定义独立 Client 接口
- 每个 Client 必须处理超时、重试、熔断、日志脱敏
- 每个 Client 必须区分可重试错误与不可重试错误
- 第三方原始响应应保留必要日志，但不得泄露密钥、Token、Cookie

### 13.3 平台发布规范

- 优先使用官方开放平台 API
- 涉及平台账号操作时，必须保留授权状态与最近刷新时间
- 发布前必须完成内容审核、素材校验、封面/标题/标签校验
- 发布结果必须以平台返回结果为准，禁止仅凭本地提交成功判断发布成功

## 14. 安全规范

### 14.1 认证与授权

- 后台管理接口必须接入 JWT 认证
- 权限模型采用 RBAC
- 平台账号、系统配置、发布控制接口必须做细粒度鉴权

### 14.2 密钥管理

- 密钥仅允许放在环境变量或密钥管理系统
- 禁止提交到 Git 仓库
- 日志中禁止输出明文密钥、令牌、Cookie、手机号、邮箱等敏感信息

### 14.3 内容安全

- 所有待发布内容必须经过敏感词过滤
- 支持人工审核开关，作为平台高风险阶段的保护措施
- 新闻来源必须保留原始链接与来源标识
- 对版权风险素材必须保留来源凭据

### 14.4 接口安全

- 登录、发布、配置修改等接口必须加限流
- Webhook/回调接口必须验签
- 关键管理操作必须写审计日志

## 15. 日志、监控与可观测性规范

### 15.1 日志规范

- 日志统一使用结构化日志
- 每条日志至少包含：时间、级别、服务名、环境、request_id
- 任务链路日志应附带：task_id、news_id、video_id、publish_record_id
- 错误日志必须包含错误码、异常摘要、重试次数

### 15.2 监控规范

必须监控以下指标：

- API QPS、响应时间、错误率
- Celery 队列长度、任务耗时、失败率、重试率
- PostgreSQL 连接数、慢查询、锁等待
- Redis 内存、连接数、阻塞情况
- 发布成功率、审核拦截率、重复抓取率

### 15.3 告警规范

以下情况必须触发告警：

- 发布成功率连续下降
- 视频生成失败率超过阈值
- 外部平台连续限流或鉴权失败
- 数据库连接耗尽
- 队列积压超过阈值

## 16. 测试规范

### 16.1 测试分层

- 单元测试：服务层、工具函数、状态机、校验逻辑
- 集成测试：数据库、Redis、OSS、第三方接口适配层
- 契约测试：前后端接口、平台回调、第三方响应解析
- 端到端测试：从新闻入库到发布记录回写的主链路

### 16.2 必测场景

- 新闻去重
- 重复任务提交
- LLM 返回异常结构
- 视频生成失败重试
- 发布失败补偿
- 平台回调重复投递
- 内容审核拒绝
- 权限越权访问

### 16.3 覆盖要求

- 核心服务层必须覆盖正常路径、异常路径、幂等路径
- 状态流转逻辑必须有完整单测
- 高风险模块测试覆盖率应高于一般模块

---

## 17. 配置与环境规范

### 17.1 环境分层

至少区分以下环境：

- `local`
- `dev`
- `test`
- `staging`
- `prod`

### 17.2 配置管理

- 所有配置统一从配置中心或环境变量读取
- 不同环境使用独立数据库、Redis、OSS Bucket、第三方账号
- 禁止测试环境调用生产平台发布能力
- 发布开关、审核开关、模拟开关必须可配置

---

## 18. 发布与上线规范

### 18.1 发布前检查

发布前必须完成：

- 数据库迁移评审
- 接口兼容性检查
- 任务幂等性检查
- 关键日志与监控埋点检查
- 敏感配置核对
- 平台沙箱或测试账号验证

### 18.2 上线要求

- 后端服务必须容器化
- 配置必须与镜像解耦
- 关键任务支持灰度和回滚
- 发布平台能力必须支持开关控制

### 18.3 回滚原则

- 优先回滚流量和开关，再回滚代码
- 涉及数据库变更时必须提前设计兼容回滚路径
- 对已触发的异步任务需评估回滚后的兼容性

---

## 19. Code Review 规范

后端 PR 合并前，至少检查以下内容：

- 是否遵守分层职责
- 是否引入硬编码配置或敏感信息
- 是否破坏幂等性与状态机约束
- 是否缺少异常处理与日志
- 是否缺少数据库索引或迁移脚本
- 是否补充必要测试
- 是否影响已有接口兼容性

---

## 20. MVP 阶段执行要求

结合需求文档中的 MVP 范围，第一阶段后端应优先保证以下能力可用：

1. AI 新闻抓取与去重入库
2. 基于新闻生成标题、摘要、脚本
3. 生成配音、字幕并完成视频合成
4. 完成抖音单平台发布闭环
5. 定时任务调度与发布记录追踪

MVP 阶段不建议过早引入复杂微服务拆分，应先在单体分层架构下把业务闭环、幂等性、可追踪性和发布安全做扎实。

---

## 21. 附录：后端开发红线

以下行为在项目中默认禁止：

- 未审核内容直接进入自动发布
- 直接在控制器中调用第三方平台 SDK
- 无幂等保护地重复提交发布任务
- 无迁移脚本直接修改数据库结构
- 无日志上下文地打印异常
- 将平台 Token、Cookie、密钥写入代码或日志
- 使用前端传入状态直接覆盖后端真实业务状态

---

## 22. 结论

本规范的核心目标不是增加流程，而是保证 EasyShorts AI 在“新闻采集 -> 内容生成 -> 视频制作 -> 自动发布 -> 数据回收”这条高风险、长链路业务中，具备可落地、可维护、可扩展、可审计的后端工程能力。

- 数据库表结构设计文档
- 状态机设计文档
- 第三方平台接入规范
- 异常码字典
