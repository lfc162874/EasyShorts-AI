## 一、文档目的

本文档只针对**后端开发计划**。  
它基于最新的 [需求文档v3](./需求文档v3.md) 和 [技术架构文档](./技术架构文档.md)，用于规划当前已经完成的基础设施之后，后续后端要实现的核心能力。

当前后端已经完成的基础能力包括：

1. 登录认证、JWT 鉴权、RBAC 权限控制
2. 用户、角色、菜单、系统配置、日志、任务基础结构
3. 来源管理、RSS / 网页采集、抓取记录、去重入库
4. 基础内容生成接口与任务底座

因此，这份计划**从热点识别阶段开始往后排**，不再重复规划已经完成的基础设施和采集底座。

## 四、后端开发边界

### 4.1 本版本后端包含

1. 热点分析服务
2. 摘要与解读服务
3. 订阅与推送服务
4. Agent 工作流和任务编排
5. 热点、推送、任务、日志、统计类接口
6. 后端测试、部署、迁移、初始化脚本

### 4.2 本版本后端不包含

1. 短视频生成链路
2. 配音、字幕、FFmpeg 合成
3. 抖音、快手视频发布
4. 前端页面设计与视觉交互
5. 复杂 BI 系统
6. 企业级多租户

### 4.3 后端开发原则

1. 先规则后智能，先把热点识别和推送链路跑稳，再逐步增强 Agent 能力。
2. 先结构化后生成式，聚类、评分、订阅匹配尽量结构化，LLM 主要负责摘要、解读和决策增强。
3. 先单渠道后多渠道，先打通一个稳定推送渠道，再扩展更多平台。
4. 所有关键结果保留结构化记录，支持追踪、重试、回放和人工复核。
5. 所有任务都要异步化，避免接口线程直接承担耗时工作。

## 六、阶段详细计划

### 6.1 阶段 3：热点识别与聚类分析

**目标：** 从采集内容中识别真正值得关注的热点，形成“内容 -> 热点 -> 聚类”的稳定分析链路。

**主要任务：**

1. 新增热点分析相关数据结构，建议补充：
   - `hot_topics`
   - `topic_clusters`
   - `topic_cluster_items`
   - `keyword_rules`
2. 实现热点评分规则：
   - 来源权重
   - 发布时间衰减
   - 关键词命中
   - 多来源重复出现
   - 用户关注关键词加权
3. 实现重复报道聚类：
   - URL / 标题归一化
   - 内容哈希去重
   - 标题相似度判断
   - 同事件多来源合并
4. 实现趋势识别：
   - 上升
   - 持续
   - 下降
5. 实现标签分类：
   - 模型
   - Agent
   - 开源
   - 融资
   - 产品发布
   - 社区热议
6. 提供热点列表、聚类详情、趋势统计相关 API。
7. 建立热点分析任务的异步执行与失败重试机制。

**交付件：**

- 热点分析服务
- 聚类分析任务
- 热点列表接口
- 热点详情接口
- 趋势与标签统计接口

**验收标准：**

- 能基于多条新闻生成热点条目
- 能按规则把重复报道归并到同一热点
- 能返回热点分数、趋势、标签和来源聚合结果
- 能通过关键词筛选热点

---

### 6.2 阶段 4：摘要解读与日报体系

**目标：** 把热点内容转化为更容易理解和消费的信息，支持单条热点解读、日报、周报和摘要输出。

**主要任务：**

1. 实现摘要生成：
   - 文章摘要
   - 热点摘要
   - 多来源摘要
2. 实现要点提取：
   - 3 到 5 条核心信息
   - 关键结论
   - 相关背景
3. 实现影响分析：
   - 对行业的影响
   - 对产品的影响
   - 对技术趋势的影响
4. 实现风险提示：
   - 争议点
   - 不确定性
   - 来源差异
5. 实现日报 / 周报生成：
   - 热点标题
   - 核心摘要
   - 重点条目
   - 影响分析
6. 实现 prompt 模板配置化，支持不同摘要风格和不同输出长度。
7. 提供摘要重试、人工修订和报告导出接口。

**交付件：**

- 摘要解读服务
- 日报生成服务
- 周报生成服务
- 摘要配置接口
- 报告导出接口

**验收标准：**

- 单条热点可生成摘要、要点与影响分析
- 多条热点可聚合生成日报内容
- 报告内容支持重复生成和人工修订

---

### 6.3 阶段 5：订阅规则与自动推送

**目标：** 根据订阅规则，把热点和报告推送到指定渠道。

**主要任务：**

1. 实现订阅规则管理：
   - 关键词
   - 来源
   - 分类
   - 热度阈值
   - 推送频率
2. 实现推送渠道管理：
   - 邮件
   - 企业微信机器人
   - 飞书机器人
   - Webhook
3. 实现推送模板管理：
   - 不同渠道不同格式
   - 日报模板
   - 即时提醒模板
4. 实现推送任务：
   - 即时推送
   - 定时摘要推送
   - 失败重试
5. 实现推送记录和状态追踪：
   - 推送时间
   - 目标渠道
   - 接收对象
   - 推送状态
   - 失败原因
6. 提供推送模拟、重发、取消和批量重试接口。

**交付件：**

- 订阅规则服务
- 推送服务
- 推送模板服务
- 推送记录接口
- 渠道配置接口

**验收标准：**

- 能按规则筛选热点并触发推送
- 至少打通一个稳定推送渠道
- 推送失败可重试、可追踪、可查看原因

---

### 6.4 阶段 6：Agent 协作与任务编排

**目标：** 把热点监控链路从“单服务调用”升级为“可编排、可观测”的智能体工作流。

**主要任务：**

1. 定义 Agent 职责边界：
   - 来源监控 Agent
   - 热点分析 Agent
   - 摘要解读 Agent
   - 推送决策 Agent
2. 使用 AgentScope 组织工作流：
   - 触发条件
   - 上下文传递
   - 结果回写
   - 失败回退
3. 建立任务编排中心：
   - 手动触发
   - 定时触发
   - 失败重放
   - 执行日志
4. 建立 prompt 配置与版本管理：
   - 摘要 prompt
   - 解读 prompt
   - 推送 prompt
   - 决策 prompt
5. 保留人工兜底入口：
   - 人工审核
   - 手动修改热点结论
   - 手动触发推送

**交付件：**

- Agent 工作流定义
- 任务编排服务
- Agent 执行日志接口
- Prompt 配置接口

**验收标准：**

- Agent 链路可单独触发并完整回写结果
- 关键任务节点可追踪、可回放、可失败重试
- 规则引擎与 Agent 决策能并存

## 八、风险与应对

| 风险 | 说明 | 应对方案 |
| --- | --- | --- |
| 来源结构不稳定 | 网站结构变化会影响采集和分析 | 规则配置化，失败记录可追踪 |
| 热点噪音过高 | 同一事件被多次重复采集 | URL 去重、标题相似度去重、聚类合并 |
| 推送打扰过多 | 用户收到过多消息 | 设计阈值、频率、分级推送和订阅过滤 |
| 分析结果波动 | LLM 结果可能不稳定 | 保留原文、保留结构化结果、支持人工复核 |
| 渠道依赖风险 | 第三方推送渠道可能不稳定 | 记录失败原因并支持重试与多渠道备用 |
| 范围继续膨胀 | 容易重新扩展到视频生成 | 严格按 v3 边界推进，视频链路不纳入本版本 |
