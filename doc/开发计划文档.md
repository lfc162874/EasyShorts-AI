## 三、V1 版本定义

### 3.1 V1 总体目标

V1 版本的目标不是先做一个“展示 Demo”，而是完成一套可运行、可追踪、可配置、可人工干预的基础业务闭环，使系统具备如下能力：

1. 能抓取和管理 AI 新闻内容。
2. 能对新闻做筛选、翻译、摘要和脚本生成。
3. 能生成配音、字幕与短视频文件。
4. 能将视频发布到目标平台，并记录发布结果。
5. 能在后台页面中完成配置、查看、审核、追踪和管理。
6. 能支撑基础日志、任务重试、系统配置和发布追踪。

### 3.2 V1 范围说明

V1 版本定义为“基础功能开发完成”，具体范围如下：

| 模块 | V1 是否纳入 | V1 目标 |
| --- | --- | --- |
| 新闻采集 | 是 | 支持数据源管理、抓取、去重、入库、定时更新 |
| 内容筛选 | 是 | 支持热度评估、基础分类、翻译、敏感词过滤 |
| 文案生成 | 是 | 支持摘要、脚本、标题、标签生成 |
| 视频生成 | 是 | 支持基础模板、字幕、配音、合成、OSS 上传 |
| 自动发布 | 是 | 至少完成抖音完整发布链路；快手纳入 V1 目标，但可按平台接入条件分阶段完成 |
| 数据分析 | 是 | 支持基础播放、互动、发布结果统计 |
| 系统管理 | 是 | 支持账号配置、数据源配置、Prompt 配置、日志查看、任务管理 |
| 审核与干预 | 是 | 支持人工审核、失败重试、状态追踪 |
| 多平台扩展 | 否 | B 站、小红书等扩展放入后续版本 |
| 多语言国际化 | 否 | 暂不作为 V1 验收前置条件 |
| 微服务拆分 | 否 | V1 采用单体服务 + 异步任务架构 |
| 高级智能编排 | 否 | 先满足稳定可用，复杂自治策略延后 |

### 3.3 V1 与 MVP 的关系

建议研发时采用“**先打通 MVP，再补齐 V1**”的方式推进：

1. MVP 阶段先打通最小业务闭环：新闻抓取 -> 文案生成 -> 视频生成 -> 抖音发布 -> 发布记录。
2. 在 MVP 闭环稳定后，补齐系统管理、基础分析、人工审核、快手接入、日志与任务治理能力。
3. 当以上能力完成后，视为 V1 达成。

### 3.4 V1 不追求的目标

以下内容不应作为 V1 上线阻塞项：

1. 多平台全量适配与统一编排中心。
2. 高级推荐策略、复杂热度模型、复杂多 Agent 自主协作。
3. 高级 BI 报表、复杂运营看板、跨平台智能归因。
4. 多租户、组织级权限、商业化结算。
5. 微服务拆分、Kubernetes 化、复杂高可用部署。

## 四、开发策略

### 4.1 总体策略

开发顺序采用“由下至上、先后端主链路、再前端完善、最后整体联调”的方式：

1. 先完成数据模型、状态机、基础配置与异步任务底座。
2. 再打通新闻采集、内容生成、媒体生成三段主链路。
3. 在此基础上接入平台发布与发布记录管理。
4. 然后补齐前端后台管理、日志监控和数据统计页面。
5. 最后进行联调、压测、异常处理与上线前收口。

### 4.2 建议研发模式

如果是单人开发，建议采用以下模式：

1. 后端主链路优先。
2. 前端以后台管理能力为目标，不做过度 UI 设计。
3. 平台发布优先做一个平台可用，再扩展第二个平台。
4. 每完成一个阶段，立刻补最基础的接口验证和状态校验。

如果是 2 到 3 人小团队，建议并行方式如下：

1. 后端负责数据模型、任务链路、平台集成。
2. 前端负责管理后台、审核与配置页面。
3. 一人兼顾联调、测试、部署与文档收口。

## 六、阶段详细计划

### 6.1 阶段 0：启动准备与方案冻结

**目标：** 将现有文档转为可执行开发基线。

**主要任务：**

1. 明确 V1 范围与非范围。
2. 确认数据库核心实体和状态流转草案。
3. 确认第三方依赖清单：LLM、TTS、OSS、平台开放能力。
4. 明确开发目录结构、分支策略、环境变量规范。
5. 明确联调方式、日志规范和错误码规范。

**交付件：**

- 开发计划文档
- 环境变量清单
- 模块拆分清单
- 数据表与状态机草案

**验收标准：**

- 研发成员对 V1 范围无歧义
- 所有基础依赖的接入前提已经确认

### 6.2 阶段 1：基础设施与底座能力建设

**目标：** 建立后端和前端的基础工程骨架，并补齐 V1 所有业务功能共享的基础设施能力。

本阶段不是简单把项目脚手架跑起来，而是要把后续所有功能都会依赖的公共底座先搭稳。  
如果阶段 1 没有完成，后续新闻、视频、发布、分析这些业务模块即使能开发，也会因为登录、权限、菜单、上传、日志、配置等能力缺失而难以联调和验收。

#### 6.2.1 阶段 1 的“基础设施”定义

本项目在阶段 1 中必须完成的基础设施包括：

1. **工程基础设施**
   包括前后端工程初始化、目录结构、环境变量规范、配置加载、本地开发环境、Docker 运行方式。
2. **身份认证基础设施**
   包括登录、退出登录、JWT 鉴权、用户状态校验、路由守卫、接口鉴权拦截。
3. **用户与权限基础设施**
   包括用户管理、角色管理、菜单管理、按钮权限、默认管理员初始化、权限校验中间层。
4. **后台导航基础设施**
   包括管理后台基础布局、侧边菜单、顶栏、面包屑、标签页或导航状态管理。
5. **文件上传基础设施**
   包括统一上传接口、文件类型和大小校验、上传记录、预览地址、OSS 存储抽象。
6. **配置中心基础设施**
   包括系统参数、Prompt 模板、数据源配置、平台账号配置、敏感词基础配置。
7. **日志与任务基础设施**
   包括操作日志、错误日志、任务日志、异步任务表、状态流转记录、Trace ID。
8. **通用接口基础设施**
   包括统一响应结构、错误码、分页结构、通用字典枚举、时间格式规范。
9. **安全基础设施**
   包括密码加密、敏感字段脱敏、基础 RBAC、接口访问控制、文件访问安全策略。

#### 6.2.2 主要任务

1. 初始化 FastAPI 工程、基础目录、配置管理、日志系统、异常处理体系。
2. 搭建 PostgreSQL、Redis、Celery、Celery Beat、本地 Docker 环境和基础启动脚本。
3. 建立核心基础数据表：`user`、`role`、`menu`、`user_role`、`role_menu`、`operation_log`、`task_job`、`file_asset`、`platform_account`、`system_config`。
4. 建立核心业务起始数据表：`news`、`video`、`publish_record`，为后续业务模块预留扩展字段。
5. 设计统一响应结构、错误码规范、分页结构、全局异常处理和请求追踪机制。
6. 完成登录、退出登录、当前用户信息获取、JWT 鉴权、接口权限拦截能力。
7. 完成用户管理、角色管理、菜单管理、按钮权限模型和默认管理员初始化逻辑。
8. 初始化 Vue 3 管理后台骨架，完成登录页、基础布局、菜单渲染、路由守卫、权限路由控制。
9. 建立全局请求封装、鉴权失效处理、全局错误提示、基础状态管理。
10. 完成统一文件上传能力，支持图片、音频、视频等基础文件类型上传，并预留 OSS 存储适配层。
11. 完成文件元数据记录、文件预览地址生成、大小与类型校验、上传失败错误处理。
12. 完成系统配置中心的基础能力，包括 Prompt 模板配置、数据源配置、平台账号配置和通用参数管理。
13. 完成操作日志、错误日志、任务日志的基础入库和后台查看能力。
14. 建立基础任务状态模型和异步任务通用执行框架，为后续抓取、生成、发布链路复用。
15. 输出 `.env.example`、本地启动说明、开发约定文档，确保新成员可快速启动环境。

#### 6.2.3 交付件

- `backend/` 基础工程
- `frontend/` 基础工程
- 数据库迁移脚本
- Docker 本地运行方案
- 登录与鉴权接口
- 用户、角色、菜单管理基础接口
- 管理后台基础布局与动态菜单框架
- 文件上传接口与文件元数据模型
- 系统配置中心基础页面与接口
- 操作日志、错误日志、任务日志基础能力
- `.env.example` 与本地启动文档

#### 6.2.4 验收标准

阶段 1 完成时，至少必须满足以下条件：

1. 前后端工程可按文档本地启动，基础环境依赖可正常连接。
2. 数据库迁移可执行，默认管理员账号可初始化。
3. 用户可通过登录页完成登录，登录态可在接口和页面路由中生效。
4. 后台基础布局、侧边菜单、面包屑和权限路由可正常工作。
5. 用户、角色、菜单三类基础管理能力可用，至少支持基础增删改查与权限绑定。
6. 文件上传能力可用，至少支持图片与视频文件上传、校验、存储记录和预览地址返回。
7. 系统配置页可维护基础参数、平台账号、数据源和 Prompt 模板。
8. 操作日志、错误日志、任务日志具备最基础查看能力。
9. 异步任务框架可创建任务、记录状态，并支持基础失败信息记录。
10. 阶段 1 完成后，阶段 2 到阶段 5 的业务模块无需再重复建设登录、权限、上传、配置和日志基础能力。

## 七、模块任务拆分建议

## 八、V1 验收标准

### 8.1 功能验收

V1 必须满足以下条件：

1. 后台可配置新闻源并定时抓取新闻。
2. 新闻可完成去重、分类、翻译和基础内容过滤。
3. 可根据新闻生成摘要、脚本、标题与标签。
4. 可根据脚本生成音频、字幕和视频成品。
5. 可将视频上传到对象存储并保留访问地址。
6. 可在后台发起发布任务并查看发布结果。
7. 可查看发布记录、任务状态、失败原因和重试结果。
8. 可配置基础系统参数、平台账号和 Prompt 模板。
9. 可查看基础运营统计数据。

### 8.2 工程验收

V1 必须满足以下工程条件：

1. 前后端工程可本地一键启动或按文档启动。
2. 核心环境变量、部署步骤、数据库迁移过程已文档化。
3. 核心接口具备基本测试覆盖。
4. 核心任务链路具备日志与失败重试能力。
5. 系统关键状态流转具备明确枚举与可追踪记录。

## 九、风险与应对

| 风险 | 说明 | 应对方案 |
| --- | --- | --- |
| 平台接入受限 | 抖音、快手开放能力与账号资质可能受限制 | 先明确接入条件，优先打通抖音，快手采用适配器设计 |
| 媒体生成链路复杂 | 配音、字幕、视频模板、FFmpeg 容易出现链路问题 | 先实现单模板稳定输出，再逐步扩展模板能力 |
| 外部依赖不稳定 | LLM、TTS、OSS、平台接口均存在波动 | 所有外部调用统一封装并加重试、超时、降级 |
| 合规风险 | 自动发布内容存在敏感内容和平台风控问题 | 加入敏感词过滤、人工审核状态、日志审计 |
| 范围膨胀 | V1 易被不断加入高级需求 | 明确 V1 非范围，未经确认不新增阻塞需求 |

## 十一、结论

本开发计划将 V1 定义为“基础功能完整闭环版本”，即：

- 不是只做一个 Demo；
- 不是一次性追求高级扩展；
- 而是优先做出一套可运行、可配置、可追踪、可维护的基础业务系统。

建议执行路径为：**先打通 MVP 闭环，再逐阶段补齐 V1 的系统管理、分析、审核和第二平台能力**。  
当阶段 0 到阶段 7 全部完成并通过验收后，可视为 EasyShorts AI 的 V1 版本开发完成。
