## 二、前端目标

### 2.1 核心目标

前端要完成的不只是“结果展示”，而是**把每一个 Agent 的处理流程可视化**。

用户在页面上需要清楚看到：

1. 是哪一条内容进入了智能处理
2. 当前执行到了哪个 Agent
3. 每一个 Agent 的输入是什么
4. 每一个 Agent 的输出是什么
5. 哪一步成功、哪一步失败、哪一步跳过
6. 最终得到的热点结果、推送决策、简报结果是什么

### 2.2 本阶段要达成的体验

用户从内容中心进入后，可以：

1. 手动发起 Agent 智能处理
2. 查看 Agent 运行列表
3. 打开某次运行详情
4. 按步骤查看 `HotspotAgent -> ClassificationAgent -> SummaryAgent -> EnrichmentAgent -> PushPlannerAgent -> DigestAgent`
5. 查看每一步的输入、输出、状态、耗时、错误信息
6. 在热点池、简报中心、推送计划中继续查看处理结果

## 三、前端设计原则

### 3.1 流程必须清晰

Agent 模块不是一个黑盒。  
页面必须明确呈现：

- 当前步骤
- 已完成步骤
- 待执行步骤
- 失败步骤

### 3.2 结果必须可追溯

每个结果都要能追溯回原始内容和具体 Agent 步骤。

### 3.3 中间产物必须可查看

不能只展示“最终摘要”或“最终推送计划”，还要能查看：

- 热度评分结果
- 分类标签结果
- 摘要结果
- 解读结果
- 推送决策结果

### 3.4 手动操作必须留入口

需要支持：

1. 重新执行整条流程
2. 重试单个步骤
3. 人工查看原文
4. 查看运行日志
5. 手动进入下一步业务页面

### 页面支持操作

## 5.2 Agent 运行详情页

**建议路由：**

- `/agent/runs/:id`

### 页面定位

这是整个 Agent 模块最关键的页面。  
它必须清晰呈现每一个 Agent 的处理流程。

### 页面布局建议

建议拆成四块：

1. 顶部运行概览
2. 中间 Agent 流程图
3. 左下步骤列表 / 时间线
4. 右下输入输出详情区

### 顶部运行概览

展示内容：

- 运行编号
- 业务类型
- 当前状态
- 当前步骤
- 总耗时
- 开始 / 结束时间
- 发起方式（手动 / 自动）

### Agent 流程图

必须把这条流程直接画出来：

```text
内容输入
  ↓
HotspotAgent
  ↓
ClassificationAgent
  ↓
SummaryAgent
  ↓
EnrichmentAgent
  ↓
PushPlannerAgent
  ↓
DigestAgent
```

每个节点要显示：

- Agent 名称
- 状态
- 耗时
- 是否可点击

### 步骤详情区

点击任意 Agent 节点后，右侧展示：

1. 本步骤输入摘要
2. 本步骤结构化输出
3. 错误信息
4. Prompt 版本
5. 模型名称
6. 开始 / 结束时间
7. 重新执行按钮

### 页面支持操作

1. 查看原始内容
2. 查看步骤日志
3. 重试整条流程
4. 重试单个步骤
5. 跳转热点详情
6. 跳转简报详情
7. 跳转推送计划详情

### 重点说明

这个页面要做到：

**用户一眼能看出每个 Agent 做了什么、产出了什么、失败在哪里。**

这也是本阶段前端开发的最高优先级。

## 六、组件规划

为了支撑“每一个 Agent 的处理流程可见”，前端建议抽离以下核心组件：

### 6.1 `AgentFlowBoard`

用于展示 Agent 流程图。

展示内容：

- 节点名称
- 节点状态
- 节点耗时
- 当前步骤高亮

### 6.2 `AgentStepTimeline`

用于展示按时间顺序的处理步骤。

### 6.3 `AgentStepDetailPanel`

用于展示当前步骤的：

- 输入
- 输出
- 日志
- 错误
- 运行信息

### 6.4 `AgentResultCard`

用于展示单个 Agent 的结果摘要。

例如：

- 热点评分卡
- 分类标签卡
- 摘要结果卡
- 推送决策卡

### 6.5 `ArtifactViewer`

用于展示结构化 JSON、文本结果和中间工件。

## 十、验收标准

满足以下条件可视为前端阶段完成：

1. 用户可以发起 Agent 智能处理任务。
2. 用户可以在页面中清楚看到每一个 Agent 的处理流程。
3. 用户可以查看每个 Agent 的输入、输出、状态、耗时和错误。
4. 用户可以从运行详情跳转到热点结果、简报结果和推送计划。
5. 页面支持整条任务重试和单步重试入口。
6. 页面在长文本、失败状态、空状态下表现稳定。

## 十一、建议开发顺序

前端建议严格按以下顺序推进：

1. 先做 `Agent 处理中心`
2. 再做 `Agent 运行详情页`
3. 再做 `热点池 + 热点详情`
4. 再做 `简报中心 + 推送计划`
5. 最后做 `配置、优化、联调`

这个顺序的关键好处是：

1. 最早就能看到每个 Agent 的流程
2. 后端只要先给运行接口，前端就能先把核心页面搭起来
3. 热点、简报、推送这些结果页都能自然挂在 Agent 运行链路之后
